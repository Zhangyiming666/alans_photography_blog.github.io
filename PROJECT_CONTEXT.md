# Alan's Photography Blog 项目说明

## 1. 项目定位

这是一个以中国行政区地图为核心入口的个人摄影作品站。  
当前项目是纯静态前端，不依赖后端服务，适合部署到 GitHub Pages、OSS 静态托管等平台。

项目的核心目标不是做“通用相册页”，而是做一张**可下钻的中国地图摄影档案**：

- 从全国视图进入省级、城市、区县
- 每个行政区可以绑定一组或多组照片
- 照片按真实区域组织，而不是手工摆点
- 页面风格偏作品集而非产品说明页

从最近的迭代方向来看，项目更偏向：

- 地图优先
- 视觉优先
- 少解释性文案
- 少“演示口吻”
- 多真实照片、多真实地区

---

## 2. 当前技术形态

项目当前是标准静态站：

- HTML：页面结构
- CSS：视觉样式、响应式布局、动效
- JavaScript：地图加载、地图下钻、相册渲染、灯箱预览、封面动效
- ECharts：地图底图与交互
- GeoJSON：行政区边界数据

当前没有：

- Node 服务端
- 数据库
- CMS
- 登录系统
- 图片上传后台

所以这个项目**本质上仍然属于静态网站**。  
如果地图线上打不开，通常不是“静态站不支持”，而是**外部 GeoJSON 数据源不可用或被拦截**。

## 4. 各文件 / 文件夹作用

### `index.html`

页面骨架文件，当前主要包含 3 个部分：

1. `#coverStory`
   - 首屏视觉封面
   - 有进入地图的按钮
   - 是“作品集开场”而不是导航页

2. `#atlas`
   - 地图主体区域
   - 左边是地图舞台
   - 右边是当前行政区对应的照片面板

3. `.lightbox`
   - 灯箱预览层
   - 点击照片后放大查看

页面底部直接加载：

1. `echarts` CDN
2. `./data/photo-catalog.js`
3. `./script.js`

这意味着项目的数据入口目前是**前端直接读取 JS 数据文件**。

---

### `styles.css`

站点的全部视觉样式基本都集中在这里。

当前视觉语言的特征：

- 柔和冷调背景
- 雾面玻璃感容器
- 深色正文 + 橙棕色强调色
- 标题使用更偏杂志/作品集的衬线气质
- 地图区和作品区是并列展示
- 支持桌面端与移动端响应式

样式里主要覆盖：

- 首屏封面
- 地图舞台
- 地图区工具栏
- 面包屑
- 右侧相册区
- 时间轴 / 快速索引
- 照片瀑布流 / 卡片
- 灯箱
- 动画和过渡

如果后续要继续“去说明化、保留地图主体”，这里会是主要调整文件之一。

---

### `script.js`

这是当前项目最核心的逻辑文件，负责把“地图、区域、照片、UI”串起来。

它大致承担 6 类职责：

#### 1. 地图数据源设置

文件开头定义：

- 本地优先：`./data/maps`
- 远程兜底：`https://geo.datav.aliyun.com/areas_v3/bound`

也就是说地图边界数据的加载策略是：

1. 先尝试读取站内本地 GeoJSON
2. 读不到再请求阿里云 DataV 的行政区边界接口

这就是为什么它依然能部署到静态站，但在部分网络环境下可能会“地图加载失败”。

#### 2. 页面状态管理

维护当前区域、地图层级、返回栈、缩放状态、缓存等信息。

典型状态包括：

- 当前行政区
- 进入路径栈
- 当前缩放比例
- GeoJSON 缓存
- 相册统计缓存

#### 3. 首屏封面逻辑

封面不是纯静态背景，而是由多行照片组成的动态视觉层。

这里会：

- 读取封面图片列表
- 生成横向流动的视觉轨道
- 跟随滚动产生位移和层次感

#### 4. 相册数据标准化

`photo-catalog.js` 中既有真实照片组，也有少量示意数据。

脚本里会对它们做统一处理，包括：

- 统一相册主键
- 统一照片字段
- 统一相机 / 镜头信息默认值
- 根据 `adcode` 判断某组作品是否属于当前区域

#### 5. 地图下钻

支持从中国进入：

- 省
- 市
- 区 / 县

地图点击后会：

1. 读取当前 feature 的 `adcode`
2. 获取下一级行政区边界
3. 更新地图
4. 更新右侧区域标题和相册内容

#### 6. 右侧相册和灯箱渲染

右侧区域会根据当前行政区筛出对应相册，并渲染：

- 相册卡片
- 主图 / 缩略图
- 标题 / 地点 / 拍摄日期
- 标签
- 灯箱放大预览

### `data/photo-catalog.js`

这是当前作品数据仓。

它通过：

```js
window.photoCatalog = { ... }
```

把所有相册挂到全局对象上，再由 `script.js` 读取。

#### 当前数据组织方式

顶层是行政区代码，例如：

- `100000`：全国
- `110000`：北京
- `140000`：山西
- `310000`：上海
- `330000`：浙江
- `440100`：广州（目前为示意图数据）
- `510100`：成都（目前为示意图数据）

#### 当前内容规模

当前文件里大约有：

- 7 个顶层行政区入口
- 25 组真实相册
- 若干示意相册

#### 两种数据形态

当前文件里混合存在两类数据：

1. **真实相册**
   - 使用 `photos`
   - 包含 `id`、`areas`、`title`、`meta`、`location`、`shotOn`、`camera`、`lens` 等字段
   - 对应真实照片资源

2. **示意相册**
   - 使用 `images`
   - 多为 `svg` 占位风格图
   - 主要出现在全国、广州、成都等入口

#### 绑定规则

每组真实相册都可以绑定多个行政区代码，例如：

- 省级
- 市级
- 区县级

这意味着：

- 同一组作品可以在更大区域被看到
- 地图下钻到更细一级时仍然能命中对应照片

如果未来要继续扩充作品，最推荐的方向仍然是：

- 保持 `adcode -> 相册数组` 的结构
- 每组相册单独维护 `photos`
- 不要再把真实照片直接硬编码到页面结构里

#### 当前标注规范

从 2026-03-25 这轮人工校正开始，项目里的照片标签不再允许“整组照片共用一套 tag 再复制到每张图”。

后续新增照片时，必须遵守下面这套规则：

- 每张照片都要单独看图后再写 `tags`
- `groupTags` 只能是该组的汇总标签，不能直接复制给所有照片
- 每张照片建议控制在 `2-4` 个 tag，宁可少写，不要臆测
- 不确定的内容不要硬写，例如不要因为同组其他照片有夕阳，就给本张也写 `日落`
- 对泛词要格外谨慎，例如：`蓝天`、`倒影`、`树木`、`建筑`、`街景`、`人群`
- 如果一组里照片差异明显，tag 也必须明显区分

已经明确踩过坑、后续要避免的情况：

- 把整组照片当成一个主题统一打 tag
- 根据文件夹名或地点名推断照片内容
- 为了“看起来完整”补上其实画面里不明显的词
- 过度使用太泛的标签，导致区分度不够

推荐工作流：

1. 先按地区 / 主题分组
2. 生成联系图或逐张预览
3. 逐张人工写 `tags`
4. 再从整组照片里提炼 `groupTags`
5. 最后再写入 `data/photo-catalog.js`

如果是让 Codex 帮忙新增照片，建议直接用下面这类 prompt：

```text
把这批照片接入地图作品集。

请严格执行：
- 必须逐张人工看图打 tag
- 不能整组复用同一套标签
- 先按地区和主题分组
- 每张图 2-4 个 tag 即可，宁少勿滥
- groupTags 只做汇总
- 对“日落/黄昏/蓝天/倒影/树木/街景/建筑/人群”这类泛词要谨慎
- 先输出每组每张照片的 tag 草案，等我确认后再改文件
```

### `data/maps/`

这是**本地行政区 GeoJSON 缓存目录**。

当前状态：

- 目录已存在
- 目前只有 `.gitkeep`
- 还没有实际的地图边界文件

这说明项目已经为“本地托管地图数据”做好了结构准备，但还没真正把 GeoJSON 放进来。

如果后续想让地图在 GitHub Pages / OSS 上更稳定，最重要的一步就是：

- 将 `100000_full.json`
- 常用省份 `xxxxxx_full.json`
- 甚至城市 / 区县 GeoJSON

放进这个目录。

这样网站就不再完全依赖第三方边界接口。

### `assets/photos/`

这是当前站点实际使用的图片资源目录。

目前这里大约有 **217 个文件**，包含：

- 真实 JPG 照片
- 少量 SVG 风格插画 / 占位图
- `story-cover/` 子目录中的封面用图

这些文件的特点：

- 路径已经适合网页直接引用
- 是站点运行时实际会访问的资源
- 文件名普遍采用原图哈希名或固定命名

#### `assets/photos/story-cover/`

这个目录用于首屏封面的图片版本。

`script.js` 中有 `getStoryCoverAsset()`，会尝试把：

- `./assets/photos/xxx.jpg`

映射成：

- `./assets/photos/story-cover/xxx.jpg`

因此这个目录的作用可以理解为：

- 为封面动效准备的轻量版 / 裁切版 / 专用版图片资源

如果某张图在 `story-cover/` 中不存在，通常需要检查封面逻辑是否仍有回退策略，避免首屏缺图。

---

### `Original photo/`

这是原始照片归档目录，不是前端运行时直接读取的目录。

当前情况：

- 一共约 **25 个拍摄主题文件夹**
- 总计约 **154 张原始照片文件**

命名方式基本是：

- `日期 + 地点 / 场景名`

例如：

- `2026.03.15 大宁公园`
- `2026.02.24 温岭东海塘`
- `2025.10.26 北京五四大街`
- `2025.07.13 大同云冈石窟`

这个目录更像：

- 原片 / 原始整理库
- 后续筛图的数据源
- 从中挑图后再复制到 `assets/photos/`

注意：

- 当前页面并不会直接读取 `Original photo/`
- 这里的内容更适合当“素材仓库”
- 如果以后引入自动化脚本，可以考虑从这里生成相册清单和拷贝目标图

## 8. 当前已知约束与风险

### 1. 地图边界仍然部分依赖远程接口

虽然逻辑已经支持本地优先，但 `data/maps/` 目前没有实际 GeoJSON。  
因此线上是否稳定，仍然受第三方边界接口影响。

### 2. 数据结构中仍混有示意内容

例如：

- `100000`
- `440100`
- `510100`

这些仍然是示意图数据，不是完整真实作品库。

### 3. 页面当前仍带有明显“首屏作品集封面”结构

本地当前版本不是只有地图，还保留了封面 section。  
如果后续方向是“只保留地图主体、压缩文字、去掉演示感”，需要继续精简 `index.html`、`styles.css`、`script.js`。

### 4. 工作区当前不是干净状态

当前 Git 工作区里已有修改：

- `index.html`
- `styles.css`
- `script.js`
- `data/photo-catalog.js`

此外 `Original photo/` 下存在大量未跟踪内容。  
后续操作时不要误删这些素材。

### 5. 工作区里有联系图缓存目录

当前根目录里还有一批为了人工校正 tag 临时生成的联系图目录：

- `.codex_contact_real/`
- `.codex_contact_real2/`
- `.codex_contact_sheets/`
- `.codex_contact_sheets_png/`
- `.codex_contact_sheets_narrow/`
- `.codex_contact_sheets_narrow_png/`

这些目录不是站点运行依赖，而是人工审图辅助产物。

可用于：

- 新一轮逐张校正 tag
- 快速回顾某组照片内容
- 在新对话里辅助接手

如果以后不再需要，可以手动清理；但在继续人工审图前，先保留会更方便。

## 11. 新对话接手规则

这条很重要：

Codex 在新开对话时**不会自动记住**这份 `PROJECT_CONTEXT.md` 的内容。  
它不是长期自动记忆，而是一个“放在工作区里的项目上下文文件”。

这意味着：

- 如果开了新对话，但没有让我去读这份文件，我不一定会自动知道当前项目细节
- 但只要你在新对话开头明确要求我“先读 `PROJECT_CONTEXT.md` 再继续”，我就能很快恢复上下文
- 这份文件的价值，是让每次新对话都能快速对齐项目状态，而不是替代真实记忆

新对话最推荐的开场 prompt：

```text
先读根目录的 PROJECT_CONTEXT.md，再查看当前项目文件，然后继续接手这个摄影地图站。
这次重点是：……
```

如果是新增照片任务，建议这样写：

```text

把 Original photo/ 里的这批新照片接入站点，并严格按文档里的规则逐张人工打 tag。
```
